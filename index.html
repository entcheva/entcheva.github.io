Falsehoods Programmers Believe

2017-01-09

I discovered this collection of blog posts: “Falsehoods Programmers Believe” through one of the resources linked in my last blog post. “Falsehoods Programmers Believe” is pretty much what it sounds like - a collection of falsehoods that programmers believe about different topics as they relate to programming. There are falsehoods on lots of categories including human identity, shopping, music, date and time, and more. The first blog post that started it all is “Falsehoods Programmers Believe About Names.”

In our communication workshop, we talked about Dale Carnegie’s How to Win Friends and Influence People - a popular self-help book about leadership and social skills. One of the “Six Ways To Make People Like You” is:

“Remember that a person’s name is, to that person, the sweetest and most important sound in any language.

“The average person is more interested in their own name than in all the other names in the world put together. People love their names so much that they will often donate large amounts of money just to have a building named after themselves. We can make people feel extremely valued and important by remembering their name.”

So if your name is the most important thing, it must be pretty annoying to be told that your name is wrong or invalid. Let’s look at two examples.

John Graham-Cumming, “Your last name contains invalid characters”

John Graham-Cumming wrote a blog post titled “Your last name contains invalid characters” that shared his frustrations with how web forms deal with his hyphenated last name. He says:

“Does the web site have any idea how rude it is to claim that my last name contains invalid characters? Clearly not. What they actually meant is: our web site will not accept that hyphen in your last name. But do they say that? No, of course not. They decide to shove in my face the claim that there’s something wrong with my name.”

“Yahoo oddly believes that I don’t know how to type my own name and decides to lowercase the C in Cumming. It’s willing to accept the hyphen but not that I know who I am.”

“There’s nothing wrong with my name […] What is wrong is that way this is being handled. If the system can’t cope with non-letters and spaces it needs to say that.”

“So, form designers: stop blaming the user for your inadequacies.”

Christopher Null, “Hello, I’m Mr. Null. My Name Makes Me Invisible to Computers”

Christopher Null writes in WIRED about his experience with web forms due to his last name.

As we know, null is a reserved string in some languages (like SQL) that is used to indicate that a data value does not exist.

Null writes that he often has problems with web forms where entering his last name “Null” will not work. In these cases, he has to come up with a workaround, like using his last name in combination with a middle name, or adding a period, as “Null.”

He also frequently gets snail mail from providers like American Express with his last name omitted, or addressed to simply “Mr.”

Falsehoods Programmers Believe About Names

Getting back to “Falsehoods Programmers Believe About Names,” probably the simplest falsehood is that people have a first and last name. Different cultures deal with names differently. It is common to have the family name go first, and it is also common for a person to have a single name, or more than two names…

Below is a brief excerpt from the list of “Falsehoods Programmers Believe About Names”:

  • People have exactly one canonical full name.
  • People have exactly one full name which they go by.
  • People’s names fit within a certain defined amount of space.
  • People’s names do not change.
  • People’s names are written in ASCII.

It should be somewhat obvious, but if someone’s name is their favorite word in any language, we should refrain from telling them their names are invalid.

A Brief Introduction to Algorithms and Big-O Notation

2016-12-15

Algorithms are a fundamental part of programming, computer science, and often, job interviews. As beginner programmers, algorithms can be daunting, but whether we realize it or not we’re already implementing them in different ways through the decisions we make in our code.

What are algorithms?

An algorithm is any repeatable process for finding the solution to a problem.

If you can outline how to get to the solution step-by-step, and if you can replicate those results, you are probably using an algorithm.

Algorithms in every day life

  • A recipe for baking cookies
  • Reorganizing your closet
  • Your morning routine

Why do algorithms matter?

There are many algorithms, and each one has a different impact on:

  • how long your code takes to run
  • how much memory it takes to run

That may not make a huge difference for our code today, but as our applications grow bigger and more complex it becomes increasingly important to assure that we’re writing code that’s efficient - that is, code that’s fast and does not take up a lot of space.

You can imagine that customers expect the websites, mobile apps, and other applications to load fast and not eat up tons of memory.

Algorithm efficiency and space/time complexity

Algorithm efficiency is measured in time and space complexity. Time complexity looks at how long an algorithm takes to solve, and space complexity is concerned with how much memory the algorithm takes to run.

We usually end up talking more about time complexity, since space is usually more readily available than patience.

Let’s tackle time complexity through a mathematical framework called Big O notation:

In simple terms:

“Big O Notation is a way of saying how much time it takes for a mathematical algorithm to run or how much memory it uses.” - Simple English Wikipedia

Three examples and their Big O time complexity:

1: Look up an element in an array:

Big O time complexity: O(1) constant
It takes the same amount of time regardless of input size

2: Iterate over an array:

Big O time complexity: O(n) linear
The time it takes corresponds to the size of the input

3: Binary search on a sorted array:

Big O time complexity: O(log n) logarithmic
The time it takes is divided by half each time

There are many more Big O notation complexities, and tons of algorithms. A handy tool for Big O is the Big O Cheat Sheet, linked below.
