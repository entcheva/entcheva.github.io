Refs, React & Redux

2017-02-07

As we expand our React skills to follow the Redux pattern, we are moving to a paradigm where we let the store manage state and move away from dealing with the state directly. However, there are a few exceptions, one of which is how to deal with user input from a form.

Do I have to put all my state into Redux? Should I ever use React’s setState()?

There is no “right” answer for this. Some users prefer to keep every single piece of data in Redux, to maintain a fully serializable and controlled version of their application at all times. Others prefer to keep non-critical or UI state, such as “is this dropdown currently open”, inside a component’s internal state.

Using local component state is fine. As a developer, it is your job to determine what kinds of state make up your application, and where each piece of state should live. Find a balance that works for you, and go with it.

As per today’s lecture, you could also approach it with the mindset that form state is something that should be managed on the application level state, and build out a form reducer that updates the current value of the form.

However, yesterday, before we really got into it in today’s lecture, I was thinking that having a child component manage state was a little bit outside the Redux mental model that I was working toward. As a result, I was thinking to approach managing form input differently, using a ref to capture the user input. For example:

In this case, I’m using a ref callback to store a reference to the DOM element, then dispatching that value directly to the store vs. managing state locally in that component.

There are tradeoffs with all three approaches. The tradeoff between the setState() approach and the refs approach is that with setState() you are tracking state with a child component, but with refs even though you are sending directly to the store, you are also directly accessing the DOM to pull that value. It’s a matter of preference.

Refs and focus()

A few other examples of when refs are useful:

  • Managing focus, text selection, or media playback.
  • Triggering imperative animations.
  • Integrating with third-party DOM libraries.
